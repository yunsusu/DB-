# 데이터 모델

**데이터를 사용하려는 목적에 맞게 정리하고 체계화해놓은 모형**

보통 데이터를 로우와 컬럼이 있는 테이블로 나누고 각 테이블 사이의 관계를 forign key 로 나타낸다

데이터 모델링을 하기 전에

먼저 기본적으로 우리가 데이터를 어떻게 인식해서 저장하고 싶은지를 정확하게 이해해야 한다.

이걸 몇개의 용어로 보면

### Entity (개체)

저장하고 싶은 데이터의 대상

쉽게 말하면 로우 하나를 말한다

일반화 해서 이야기 할때는

Entity Type : 일반화한 Entity 종류

라는 표현을 한다. (테이블 전체)

### Attribute (속성)

Entity에 대해서 저장하려는 내용

학생이라는 Entity라고 생각해보면

그 학생의 학번, 이름, 성별, 전공 등이 Attribute가 되는 것이다.

쉽게 컬럼

### Relationship (관계)

Entity들 사이 연결점

### Constraint (제약 조건)

일종의 규칙들

### 데이터 모델링 목적

- 저장하고자 하는 데이터에서 Entity, Attribute, Relationship, Constraint 파악
- 데이터베이스를 구축할 때 기반이 될 모델 만들기

# Relational 모델

데이터를 로우와 컬럼으로 이루어진 테이블을 가지고 표현한 모델을 말한다

여기서 이름에 들어가는 **relation은 테이블들 사이에 맺어지는 연결 관계를 의미하지는 않는다.**

이는 ‘테이블’을 의미하는 수학적 표현 이다.

즉 Relational Model은

데이터를 relation, 즉 테이블로 정리해서 표현한 모델 이다.

테이블들 사이에서 관계는 foreign key로 만들어 주는거다.

Relation Model은 우리에게 가장 익숙하고 실제 데이터베이스를 만들 때도 편리하긴 한데 몇가지 단점 때문에 데이터의 구조와 논리를 직관적으로 이해하기 힘들 수도 있다.

단점

1. 로우가 있다는 점
2. 테이블들 사이에 맺어지는 관계들의 특성을 한눈에 알기 힘들다는 점

# 데이터 모델 스펙트럼

데이터 모델은 얼마나 자세하게 표현됐는지에 따라 세 종류로 구별된다.

### 개념 모델 (Conceptual Model)

가장 추상적인 모델은 개념 모델(Conceptual Model)

개념 모델은 주로 큼지막한 entity와 간단한 연결 관계만을 나타낸다.

attrivute의 내용은 구체화하지 않고 대략적인 구조를 파악할 때 주로 사용

![스크린샷 2024-07-21 오후 7.18.05.png](https://prod-files-secure.s3.us-west-2.amazonaws.com/97725b2e-027c-4697-871f-60f9a007ed7d/7bafe0ad-f0bd-4f89-9384-2cc8322b09f7/%E1%84%89%E1%85%B3%E1%84%8F%E1%85%B3%E1%84%85%E1%85%B5%E1%86%AB%E1%84%89%E1%85%A3%E1%86%BA_2024-07-21_%E1%84%8B%E1%85%A9%E1%84%92%E1%85%AE_7.18.05.png)

### 논리 모델 (Logical Model)

조금 더 구체적인 논리 모델

이 모델은 entity뿐만 아니라 attribute들까지 표현을 하고

이 중에서 primary key, foregin key가 뭔지 표시한다.
